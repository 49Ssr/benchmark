# 005 - Lifecycle and Validation Plan Cross-Check

## Purpose

This pass checks the proposed `prepare -> run -> validate` direction against:

- the code currently in the repository
- the earlier benchmark research
- Google Benchmark behaviour
- SPEC CPU validation rules
- LLVM benchmarking guidance
- experimental-methodology research on measurement bias and statistical uncertainty

No C++ source code was changed during this review.

The goal is to make the next code change smaller and more defensible before implementation begins.

---

## 1. Verdict

The previous plan was directionally correct:

- preparation should be outside the measured interval when the benchmark claims to measure update work only
- the benchmark output must remain observable
- output must be checked after execution
- raw samples and test conditions must eventually be preserved
- hardware-specific paths must not be added before correctness can be compared

However, the earlier plan still needed several corrections.

The biggest correction is this:

> A checksum is evidence that an output existed. It is not automatically proof that the output is correct.

The next implementation must therefore separate three ideas:

1. **invariants** - basic rules that must always hold
2. **fingerprint** - deterministic evidence describing the final state
3. **reference comparison** - evidence that another implementation produced an acceptable result

The previous `setup()` name is also too vague. The runner will call the method before every measured sample, so `prepareRun()` states the real behaviour more clearly.

---

## 2. Cross-check against established benchmark behaviour

## 2.1 Separating preparation from timed work is valid

Google Benchmark supports setup/teardown callbacks and also allows timers to be paused around work that must happen for every iteration but should not be measured.

Its documentation also warns that repeatedly pausing and resuming timing has overhead. Our runner already measures one complete workload invocation at a time, so an explicit lifecycle is simpler:

```text
prepareRun()
start timer
run()
stop timer
validate()
```

This is a reasonable design for this project.

### Adjustment

Use `prepareRun()` rather than generic `setup()`.

The contract must state:

- it is called before every warm-up run
- it is called before every measured run
- it must restore equivalent starting state
- its time is not included in the benchmark sample

Without that rule, repeated samples could begin from different simulation states.

---

## 2.2 The timed boundary defines the workload

Moving allocation and initialization outside the timer changes what the benchmark measures.

The current benchmark measures approximately:

```text
allocate storage
initialize one million entities
perform 100 update steps
```

The proposed benchmark measures:

```text
perform 100 update steps on prepared entity data
```

Both can be useful, but they answer different questions.

### Adjustment

The first refactored workload should be named and versioned as an **update benchmark**, not a generic end-to-end simulation benchmark.

Possible future categories:

```text
simulation_update_prepared
simulation_end_to_end
```

The first category excludes allocation and initialization. The second may deliberately include them later.

Changing the boundary requires a workload-version change.

---

## 2.3 Preparation outside timing does not create a neutral cache state

Initializing the entire vector immediately before timing:

- commits memory pages
- writes the entity data
- warms parts of the memory hierarchy
- establishes a repeatable prepared state

It does not represent a cold-start workload.

For one million entities, the working set is much larger than a small private cache, and 100 update passes will quickly dominate the first-pass state. Even so, the starting-state policy must be stated rather than ignored.

### Adjustment

Record the initial policy as:

```text
state_policy = prepared
```

Do not describe the first refactored benchmark as a cold-cache test.

A cold-start or cache-state experiment can be added later as a separately labelled workload.

---

## 2.4 Validation must compare against something

SPEC CPU requires outputs to match expected outputs, normally within benchmark-defined tolerances.

A fingerprint generated by our program is useful, but these are not equivalent claims:

```text
Fingerprint: 0x1234
```

and:

```text
Fingerprint matches the expected reference: pass
```

A stable wrong implementation can produce a stable fingerprint.

### Adjustment

Validation should use layers.

### Layer 1 - invariants

For the current simulation, useful checks include:

- entity count remains correct
- no position, velocity or acceleration is NaN or infinite
- final `y` values are not below the collision floor
- fixed configuration values were not changed unexpectedly

These checks can catch broad failures without requiring bit-identical floating-point output.

### Layer 2 - full-state fingerprint

Read every final entity field after timing and combine its bit representation into a deterministic 64-bit fingerprint.

The fingerprint must leave the benchmark object by being stored in the result and eventually printed or exported. Otherwise the compiler may still determine that the final state is unobserved.

The fingerprint is not cryptographic. Its purpose is regression evidence and cross-path comparison.

### Layer 3 - reference comparison

For the current single-thread and two-thread implementations, each entity executes the same arithmetic in the same per-entity order. The two paths should therefore produce the same fingerprint under the same build and floating-point policy.

Later SIMD/FMA paths may produce small floating-point differences. Those paths will need explicit absolute/relative tolerances or error metrics rather than blindly requiring the same bit pattern.

---

## 2.5 Avoid blessing one large implementation as automatically correct

Using the first full-size single-thread fingerprint as a permanent golden value could preserve a bug.

### Adjustment

Add a small, non-timed correctness test later using a handful of entities and a small number of steps whose expected values can be independently calculated and reviewed.

The full benchmark then uses:

- the small correctness test to check the equations
- invariants to reject obvious failures
- a full-state fingerprint to compare implementations and detect regressions

This is stronger than relying on one opaque checksum alone.

---

## 2.6 Anti-optimization barriers are not correctness tests

Google Benchmark provides `DoNotOptimize()` and `ClobberMemory()`, but its documentation explicitly warns that `DoNotOptimize()` does not prevent all optimization of the expression.

Therefore:

- `volatile` should not be treated as a general benchmark-validation system
- a compiler barrier should not replace result validation
- the benchmark should produce real output that leaves the timed workload

### Adjustment

The simulation's final-state fingerprint and validation result should be stored in each sample.

The current volatile integer loop remains only a smoke test until it is replaced or honestly renamed.

---

## 2.7 Validation outside the timer can still affect later samples

A full scan of roughly 24 MB of entity data after every timed run is not free. Although it is excluded from the sample time, it can affect:

- cache state
- memory traffic
- temperature
- frequency before the next run

This does not mean validation should be skipped. Correctness has priority.

### Adjustment

Use a fixed sequence for every sample:

```text
prepareRun
measured run
full validation/fingerprint
record sample
```

Then later:

- measure validation cost separately
- add warm-ups
- interleave benchmark variants
- preserve raw sample order

If validation cost becomes large enough to distort the session, the project can split full reference verification from lighter per-sample evidence. That should be based on measurement, not assumed now.

---

## 2.8 Current two-thread timing includes thread startup and shutdown

The two-thread implementation creates two `std::thread` objects inside `run()` and joins them before `run()` returns.

Therefore its measured time includes:

- thread construction
- operating-system scheduling
- the entity updates
- thread joining

That is a valid end-to-end parallel-dispatch measurement, but it is not pure steady-state worker throughput.

### Adjustment

Do not silently describe the current two-thread result as pure compute scaling.

The first refactor should preserve the existing behaviour and label the policy:

```text
thread_policy = spawn_per_run
```

Later add a separate experiment with persistent worker threads created outside the timed region:

```text
thread_policy = persistent_workers
```

The two policies answer different questions and should remain separately reported.

---

## 2.9 Current thread independence remains valid but must be part of the workload definition

Each thread owns a disjoint half of the entity array. Entities do not interact, so no barrier is required between simulation steps.

This would become invalid if future work adds:

- collisions between entities
- neighbourhood lookups
- shared forces modified each step
- any cross-range dependency

### Adjustment

Record entity independence as a workload property. Do not reuse the current threading scheme automatically after changing the physics.

---

## 2.10 Repetition requires more than summary statistics

Google Benchmark supports warm-up periods, repeated runs, mean/median/deviation reporting and random interleaving because system state can change during a session.

LLVM's guidance also distinguishes low noise from low bias. Mytkowicz et al. demonstrated that apparently harmless experimental details can reverse conclusions. Kalibera and Jones further argue that uncertainty and multiple sources of non-determinism must be considered when comparing performance changes.

### Adjustment

The project should eventually report not only a per-variant summary, but also the uncertainty of comparisons between variants.

Near-term order:

1. preserve raw samples
2. add warm-ups
3. alternate or randomize variant order
4. add median, sample standard deviation and coefficient of variation
5. later add effect size and confidence intervals for comparisons

Do not claim that a tiny speed difference is meaningful merely because two averages differ.

## 3. Revised lifecycle contract

The next interface direction should be conceptually:

```cpp
class Benchmark {
public:
    virtual ~Benchmark() = default;
    virtual void prepareRun() = 0;
    virtual void run() = 0;
    virtual ValidationResult validate() const = 0;
    virtual std::string getName() const = 0;
    virtual int getWorkloadVersion() const = 0;
};
```

This is still a plan, not code currently present in the repository.

### Why each method exists

`prepareRun()`

- restores the starting state before every sample
- is not timed
- may lazily allocate storage on the first call
- must initialize equivalent inputs for all compared implementations

`run()`

- contains only the work named by the workload definition
- is the only part wrapped by the timer

`validate()`

- runs after timing
- checks invariants
- produces a full-state fingerprint
- reports whether the output is acceptable

`getWorkloadVersion()`

- prevents results from different workload definitions being silently mixed

## 4. Validation result direction

A bare boolean is too weak, while a large generic validation framework is unnecessary now.

The minimum useful model should distinguish unvalidated work from passed work.

Conceptually:

```cpp
enum class ValidationStatus {
    NotChecked,
    Passed,
    Failed
};

struct ValidationResult {
    ValidationStatus status;
    std::uint64_t fingerprint;
};
```

Possible later fields:

- diagnostic message
- maximum absolute error
- maximum relative error
- number of failed elements
- reference implementation name

Do not add those fields until a real workload requires them.

### Particle smoke-test consequence

The current integer loop should return `NotChecked` rather than pretending it passed meaningful validation.

That keeps the framework honest while the smoke test remains in the project.

---

## 5. Benchmark sample direction

The existing `BenchmarkResult` contains only name and elapsed milliseconds.

The next useful sample needs enough information to preserve what happened:

```text
benchmark name
workload version
run index
elapsed time
validation status
fingerprint
thread policy
```

Hardware and session metadata should still be stored separately later, not copied into every benchmark class.

## 6. Revised coding sequence

## Immediate patch - lifecycle and validation only

1. Add `ValidationStatus` and `ValidationResult`.
2. Add `prepareRun()`, `validate()` and `getWorkloadVersion()` to the benchmark contract.
3. Update all current benchmark classes so the project still builds.
4. Move each simulation vector into its benchmark object.
5. Make `prepareRun()` resize when necessary and restore every entity to the same initial values.
6. Keep the existing update equations unchanged in `run()`.
7. Add invariant checks and one full-state fingerprint pass in `validate()`.
8. Store validation status and fingerprint in the returned result.
9. Confirm the single-thread and two-thread fingerprints match.
10. Rename or label the workloads so the prepared-state and spawn-per-run policies are visible.

## Second patch - measurement protocol

1. Add warm-up runs.
2. Preserve run index and sample order.
3. Alternate or randomize single-thread and two-thread measured runs.
4. Move summary calculation and console printing out of the runner.
5. Add median, sample standard deviation and coefficient of variation.

## Third patch - reproducibility and export

1. Track a shared Release `CMakePresets.json`.
2. Record compiler/build/workload metadata.
3. Export raw samples to session CSV files.

Only after these stages should the first AVX2 dispatch experiment begin.

---

## 7. What this review changes from document 004

Document 004 remains broadly correct, but this pass makes these specific adjustments:

- rename `setup()` to `prepareRun()` and define it as per-sample
- treat checksum, correctness and reference comparison as separate ideas
- add a small independent correctness test to avoid blessing a large buggy reference
- explicitly label the first refactored workload as prepared-state update timing
- preserve current thread creation timing, but label it `spawn_per_run`
- plan a separate persistent-worker result later
- acknowledge that validation outside the timer can still affect later samples
- postpone confidence intervals, but record them as the correct future comparison method
- do not let the particle smoke test report a fake validation pass
